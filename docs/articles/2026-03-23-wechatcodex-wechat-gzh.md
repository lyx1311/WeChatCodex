# 我把 Codex 接进了微信，刚刚把 WeChatCodex 开源到了 GitHub

先说明一下，这个项目是受到《在微信里使用 Claude Code，刚刚在 GitHub 上开源了这个 Skill。》这篇文章的启发做出来的。

我看完那篇文章后的第一反应很直接：

既然 Claude Code 能通过微信协议层接进来，那 Codex 能不能也直接接进微信？

答案是可以。

而且不需要 OpenClaw Gateway，不需要公网 IP，不需要域名，本地跑一个 Node.js 常驻进程就够了。

我把这套方案整理成了一个开源项目：WeChatCodex。

开源地址：

https://github.com/DrDavidDa/WeChatCodex

## 01 这次我到底做了什么

简单说，就是把个人微信接到本机的 `codex` CLI。

手机微信发来一条消息，本地进程通过微信 ilink / ClawBot 协议收下来，然后把内容转成 `codex exec` 或 `codex exec resume`，等 Codex 处理完，再把结果发回微信。

整条链路是这样的：

`微信 -> 微信协议层 -> 本地 Node.js daemon -> Codex CLI -> 微信`

这里最关键的一点，是我没有走 OpenClaw Gateway 这条路，而是直接在本机把消息处理层替换成了 Codex。

这样做有几个好处：

- 部署足够轻，不需要再配公网服务
- 数据路径更短，调试起来直接
- 更适合个人电脑本地常驻使用
- 你在地铁上用手机发条消息，回到电脑前，任务可能已经跑完了

## 02 参考文章给了我什么启发

那篇文章最有价值的地方，不只是“可以在微信里聊天”，而是它把一个关键思路说透了：

微信 AI 入口的关键不在 OpenClaw Gateway 本身，而在微信 Bot 那套协议层。

只要把长轮询、扫码绑定、消息收发、媒体下载这些协议流程打通，后面的执行层其实是可以替换的。

参考文章里替换的是 Claude Code。

我这次替换的是 Codex CLI。

这两者的工程差异也很明显。

Claude Code 方案可以走 Agent SDK，有流式回调和更细的交互接口；Codex 这边我采用的是 `codex exec` / `codex exec resume` 的本地桥接方式，所以权限控制也从“逐工具 yes/no”改成了显式模式切换：

- `plan`
- `workspace`
- `danger`

这更符合 Codex 当前的运行模型。

## 03 我是怎么把这个项目做出来的

这次项目我还是按 Superpowers 的方式推进。

先 Brainstorming，把架构边界说清楚。

比如：

- 是继续借 OpenClaw Gateway，还是直接本地化
- 微信命令层要不要保留
- 图片消息怎么转给 Codex
- Codex 没有 Claude 那种权限审批回调，交互应该怎么设计

这些边界如果一开始不想清楚，后面越写越容易乱。

方案定下来之后，再写 plan，把模块拆开：

- 微信协议层
- 会话持久化
- Codex 桥接层
- 图片处理
- slash 命令
- daemon 管理

这一步的价值很大，因为它会逼着你把接口先定义清楚，再进入实现。

## 04 中间踩过的一个坑

这个项目不是一把过的。

刚开始的版本，微信里第一条消息能正常回复，但第二条开始就没反应了。

后面排查发现，问题不在微信协议，而在守护进程模型：

我一开始把 `codex exec resume` 放在了轮询线程里同步等待。

只要 Codex 处理时间一长，整个消息轮询就被卡住了。微信那边看起来就像“除了第一条，后面都没反应”。

后面我把这块改成了后台异步处理，同时给 Codex 子进程加了超时保护，并在 daemon 启动时把遗留的 `processing` 会话状态重置成 `idle`。

修完以后，整个桥就稳定了很多。

这个细节我觉得挺有代表性：

很多这类“消息进来但不回”的问题，看起来像协议层故障，实际上根因可能是本地执行层把轮询线程堵死了。

## 05 现在这个项目能做什么

目前 WeChatCodex 已经支持这些能力：

- 微信文字消息接入本机 Codex
- 多轮上下文续聊
- 图片转给 Codex 分析
- `/status`、`/clear`、`/cwd`、`/model`、`/mode` 等命令
- macOS `launchd` 常驻
- 本地日志、配置、账号和会话持久化

也就是说，它已经不是一个 Demo 了，而是一套能实际跑起来的本地桥接服务。

## 06 为什么我把它开源

原因很简单。

这类项目一旦只停留在本机目录里，价值是很有限的。

真正有意思的地方，是把方案、代码、结构和踩坑过程一起公开出来。

别人可以直接拿去用，也可以继续往下改：

- 接更多命令
- 做更细的权限模型
- 优化消息队列和并发处理
- 支持更多平台上的守护进程管理

开源之后，这个项目才会从“我本地能跑”变成“别人也能复用的方案”。
