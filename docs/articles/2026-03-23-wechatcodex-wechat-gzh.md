# 我把 Codex 接进了微信，也把它开源到了 GitHub

这篇东西的起点，确实是受到了《在微信里使用 Claude Code，刚刚在 GitHub 上开源了这个 Skill。》的启发。但我这次真正想做的，不是复刻一个 Claude 版本，而是把这条链路改造成更适合 Codex 的工程方案。

项目已经开源：

https://github.com/DrDavidDa/WeChatCodex

## 01 先说结果

现在我可以直接在微信里给本机上的 Codex 派活了。

手机发消息过来，本地 daemon 收到，转成 `codex exec` 或 `codex exec resume`，执行完以后再把结果回到微信里。

整条链路是：

`微信 -> 微信协议层 -> 本地 Node.js daemon -> Codex CLI -> 微信`

这里我没有接 OpenClaw Gateway，也没有配公网 IP，更没有加域名。

整个项目的出发点很简单：

我想把 AI 编程助手真正变成一个随手可用的入口，而不是只能坐在电脑前打开终端才能用。

## 02 这件事为什么值得做

我一直偏爱这种“入口先打通，再把工程收敛干净”的项目。

先让它活起来，再让它稳定，再让别人也能复用。

这也是我做很多工具类项目的一贯方式。

不是先堆概念，不是先做一套很漂亮的架构图，而是先把链路打通，看到真实数据流，看到真实问题，再决定哪里该抽象，哪里该克制。

WeChatCodex 也是一样。

它的核心价值，不是“微信也能聊天”这么简单，而是把一个很自然的个人工作流补齐了：

- 你在路上想到一个需求，直接微信发过去
- 本机上的 Codex 开始处理
- 到电脑前时，结果可能已经回来了

这和传统“远程桌面控制电脑”不是一回事。

我想做的是把 Codex 本身变成一个低摩擦入口，而微信刚好是那个几乎没有学习成本的外壳。

## 03 我这次怎么做的

方案上我选得很直接：

不绕 Gateway，直接复用微信 ilink / ClawBot 这一层协议。

因为只要扫码绑定、长轮询、消息格式、媒体下载这些底层链路打通，后面真正的执行器是谁，其实是可以替换的。

我这次替换成的是 Codex CLI。

所以项目最终落成了几个非常实际的模块：

- 微信协议层
- 本地守护进程
- Codex 桥接层
- 会话持久化
- 图片处理
- slash 命令系统

整个结构没有故意做得很花。

我更在意的是：

- 出问题时能快速定位
- 守护进程能稳定常驻
- 多轮上下文能接得住
- 图片和文本都能走通

## 04 这不是“写完就完了”的那种项目

这种桥接项目，真正麻烦的地方从来不是第一版写出来，而是你以为它通了，结果一上真实消息就开始暴露边界问题。

比如我这次就踩了两个很典型的坑。

第一个坑，是消息轮询被卡死。

一开始第一条消息能回，第二条开始就没反应。最后查出来，不是微信协议有问题，而是我把 `codex exec resume` 放在轮询线程里同步等待了。

只要 Codex 忙一点，整条消息轮询就会被堵住。

这个问题修法也很明确：

- 把 Codex 执行放到后台异步
- 给子进程加超时保护
- daemon 启动时清理遗留的 `processing` 状态

第二个坑，是图片消息的结构和我最初假设的不一样。

我原来按旧结构去取 `image_item.cdn_media`，但真实线上图片事件返回的是 `image_item.media`。结果就是：图片其实已经进来了，但下载阶段被当成空值丢了，最后用户侧看到的体验就会像“我明明发了图，你却说没收到”。

这种问题很有意思。

因为它不是没收到，而是收到了以后，在协议适配层被错判了。

我把这层兼容补上以后，图片链路才算真正闭环。

## 05 现在 WeChatCodex 已经能做什么

目前这套东西已经不是一个演示级别的小样了，而是能实际跑的本地服务。

它现在已经支持：

- 微信文字消息接入本机 Codex
- 基于 `thread_id` 的多轮上下文
- 图片消息转给 Codex 分析
- `/status`、`/clear`、`/cwd`、`/model`、`/mode` 等命令
- macOS 上用 `launchd` 常驻
- 本地保存账号、配置、日志和会话

我还专门把一些体验层的问题收了一遍。

比如以后如果图片真的收到了，但下载或解密失败，系统会明确告诉你“图片已收到，但下载或解密失败”，而不是再回一个误导性的“没有收到图片”。

我不喜欢那种看起来能用、但一碰真实场景就开始说瞎话的工具。

## 06 为什么我要把它开源

因为这类项目只留在我自己的电脑里，价值是不完整的。

真正有价值的是三件事一起公开：

- 思路
- 代码
- 踩坑过程

别人可以直接拿去跑，也可以继续往下改。

比如你完全可以继续加：

- 更细的权限模型
- 更稳的消息队列
- 更完整的图片 / 文件支持
- Linux 上更完整的守护进程验证

我一直觉得，好的开源不只是“把代码传上去”，而是把别人二次接手这件事的成本降下来。

这也是我做这类项目时最在意的一点：

先把真实链路跑通，再把工程打磨到别人也能复用。

WeChatCodex 只是这条路上的一个开始，后面我还会继续往下做。
